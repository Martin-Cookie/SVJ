## KRITICKE

### K1: compute_nav_stats() se vola na KAZDEM platebnim endpointu — 36x redundantne

- **Severity:** KRITICKE
- **Modul:** Platby (vsechny sub-moduly)
- **Pohled:** Performance analytik
- **Co a kde:** Kazdy endpoint v platebnim modulu vola `compute_nav_stats(db)` — 7 separatnich SQL dotazu (PrescriptionYear, VS count, Prescription count, BankStatement count, Payment agregace, debtor count, Settlement counts). Toto se vola pri KAZDEM pozadavku vcetne HTMX partial odpovedi pro hledani a razeni. Uzivatele typicky prochazi 5-10 stranek plateb za minutu = 35-70 zbytecnych SQL dotazu na navigacni karty.
- **Dopad:** Zbytecne zatizeni databaze. Pri 500+ predpisech a 1000+ platbach muze `_count_debtors_fast()` trvat stovky ms (iterace pres vsechny jednotky). Kazde stisknuti klavesy v hledani spusti kompletni recount.
- **Reseni:** (A) Cache `compute_nav_stats()` vysledek na urovni requestu nebo s krátkym TTL (5-10s). (B) HTMX partial odpovedi NEPOTREBUJI nav stats — prestat predavat `**compute_nav_stats(db)` do partial kontextu. (C) Predat nav stats jen pri full-page renderingu.
- **Varianty:** Varianta B je nejjednodussi — v partial sablonach se nav karty nerenderuji, takze nav stats nejsou pouzity. Staci pridat podmínku: `nav = compute_nav_stats(db) if not is_htmx_partial(request) else {}`.
- **Kde v kodu:** `app/routers/payments/prescriptions.py`, `statements.py`, `overview.py`, `settlement.py`, `symbols.py`, `balances.py` — vsude kde je `**compute_nav_stats(db)` v kontextu
- **Narocnost:** Nizka ~20 min
- **Zavislosti:** —
- **Regrese riziko:** Nizke — partial sablony nav karty nepouzivaji
- **Rozhodnuti:** 🔧 jen opravit
- **Jak otestovat:** Merit pocet SQL dotazu pred a po (logging). Hledat v predpisech → mene dotazu.
- **Mockup:**
  ```
  Soucasny stav:
  # Kazdy endpoint (vcetne HTMX partial):
  ctx = {
      ...
      **compute_nav_stats(db),  # 7 SQL dotazu VZDY
  }

  Navrhovany stav:
  nav = {} if is_htmx_partial(request) else compute_nav_stats(db)
  ctx = {
      ...
      **nav,
  }
  ```

### K2: Smazani roku predpisu kaskadove rozbije VS mapovani a vyuctovani bez varovani

- **Severity:** KRITICKE
- **Modul:** Platby — predpisy
- **Pohled:** Error recovery, Data quality
- **Co a kde:** Endpoint `predpisy_smazat` (`prescriptions.py:443-454`) smaze `PrescriptionYear` s cascade na vsechny `Prescription` a `PrescriptionItem`. ALE: (1) VS mapovani (`VariableSymbolMapping`) vytvurena pri importu predpisu zustanou "osirely" — odkazuji na jednotky ale ztrati se kontext odkud prisly. (2) `Settlement` zaznamy se nepregenruji — zobrazuji stare data. (3) `_count_debtors_fast()` pouziva predpisy — po smazani dluznici zmizi z dashboardu. Uzivatel nedostane ZADNE varovani o techto nasledcich.
- **Dopad:** Po smazani roku predpisu: (a) Matice plateb a dluznici se zobrazi prazdne. (b) Vyuctovani ukazuji zastarale castky. (c) Automaticke parovani selze (neni predpis pro porovnani). Uzivatel muze smazat predpisy v dobre vire ze je reimportuje, ale mezidobi rozbije vsechny navazne vystupy.
- **Reseni:** V `data-confirm` dialogu na strance `predpisy_detail.html:21` zobrazit pocty nasledku: "Smazani predpisu pro rok {rok} odstrani i {N} polozek. Vyuctovani a dluznici budou nepresne az do reimportu." Alternativne: nabidnout "Prepsat" misto "Smazat" (import noveho souboru pro stejny rok uz toto resi).
- **Varianty:** (A) Vylepsit confirm dialog s pocty, (B) Skryt mazeni za admin-only akci, (C) Automaticky pregenerovat vyuctovani po smazani.
- **Kde v kodu:** `app/routers/payments/prescriptions.py:443-454`, `app/templates/payments/predpisy_detail.html:21`
- **Narocnost:** Nizka ~15 min (varianta A)
- **Zavislosti:** —
- **Regrese riziko:** Nizke
- **Rozhodnuti:** ❓ potreba rozhodnuti — jak moc varovani pridavat
- **Jak otestovat:** Na detailu predpisu kliknout "Smazat rok" → dialog by mel ukazovat nasledky.
